# 오브젝트 3, 4장 정리 (스터디 Week2)

## 전체 핵심 요약

3장은 객체지향 설계의 출발점을 협력으로 본다.\
좋은 설계는 객체를 먼저 나누는 것이 아니라, 어떤 메시지가 오가야
하는지부터 생각한다.

4장은 데이터 중심 설계의 한계를 보여준다.\
객체를 상태 묶음으로 취급하고 외부에서 값을 꺼내 조작하면 캡슐화가
무너지고, 변경이 서비스와 컨트롤러에 몰린다.

설계는 다음 순서로 생각한다.

1.  어떤 협력이 필요한가
2.  어떤 메시지가 필요한가
3.  누가 그 메시지를 처리하는가
4.  상태와 구현은 마지막에 결정한다

------------------------------------------------------------------------

# 3장 역할, 책임, 협력

## 3장의 핵심 주제

객체는 고립된 존재가 아니라 협력 안에서 의미를 가진다.

중요한 것은 클래스 이름이 아니다.\
어떤 책임을 맡고 어떤 메시지에 응답하느냐가 더 중요하다.

3장에서 계속 강조하는 질문은 다음과 같다.

누가 이 일을 해야 하는가

좋은 답은 항상 "그 정보를 가장 잘 알고, 그 규칙을 지켜야 하는 객체"에
가깝다.

------------------------------------------------------------------------

## 핵심 개념 1 협력이 객체를 만든다

객체지향 설계를 시작할 때 흔히 하는 실수

-   엔티티부터 나눈다
-   테이블 구조부터 맞춘다
-   필드를 먼저 정리한다

3장은 반대로 말한다.

먼저 협력을 본다.\
예를 들어 예약 시스템이라면

-   예약 가능 여부를 판단해야 한다
-   결제 상태를 바꿔야 한다
-   취소 후 후속 처리를 해야 한다

이 협력을 따라가다 보면 필요한 객체와 책임이 드러난다.

------------------------------------------------------------------------

## 핵심 개념 2 책임은 행동 중심으로 본다

좋은 책임은 "무엇을 안다"보다 "무엇을 한다"로 표현된다.

예

-   재고를 예약한다
-   결제를 승인한다
-   예약을 취소한다
-   통계를 갱신한다

반대로 좋지 않은 책임 표현

-   재고 수량 값을 가진다
-   상태 필드를 보관한다
-   결제 완료 시간을 저장한다

상태는 책임 수행을 위한 수단이다.

------------------------------------------------------------------------

## 핵심 개념 3 책임은 정보를 가진 객체에게 간다

자신의 상태와 규칙을 가장 잘 아는 객체가 직접 행동해야 한다.

예

`RoomInventory`\
→ 잔여 수량과 마감 여부를 안다

`TossPayment`\
→ 결제 상태 전이를 안다

`ReservationEmailOutbox`\
→ 처리 완료 여부를 안다

이 책임을 서비스가 대신 수행하면 객체는 수동적인 데이터 구조가 된다.

------------------------------------------------------------------------

## 핵심 개념 4 메시지가 먼저다

협력은 메시지로 이루어진다.

중요한 질문

-   어떤 메시지가 필요한가
-   누가 그 메시지에 응답하는가

예

-   `reserve()`
-   `cancel()`
-   `approve()`
-   `markAsProcessed()`

이런 메시지는 객체가 수행할 역할을 분명하게 드러낸다.

------------------------------------------------------------------------

## 핵심 개념 5 역할은 협력을 단순하게 만든다

여러 객체가 비슷한 방식으로 협력할 수 있다면, 구체 클래스보다 역할이
더 중요해진다.

실무에서는 완전한 인터페이스가 없어도 역할 관점으로 설계를 읽을 수 있다.

예

-   예약 가능한 대상
-   취소 가능한 결제
-   집계 가능한 통계

역할 관점으로 보면 서비스는 모든 세부 구현을 알 필요가 없다.

------------------------------------------------------------------------

## 3장 핵심 정리

좋은 협력 구조의 특징

-   객체가 메시지에 응답해 스스로 행동한다
-   책임이 서비스 하나에 몰리지 않는다
-   상태 변경 규칙이 해당 객체 안에 모인다
-   메시지 이름만 봐도 도메인 의도가 읽힌다

------------------------------------------------------------------------

# 4장 설계 품질과 데이터 중심 설계

## 4장의 핵심 주제

겉보기에는 객체가 많아 보여도, 실제로는 데이터 중심 설계일 수 있다.

데이터 중심 설계의 전형적인 흐름

1.  객체를 필드 묶음으로 만든다
2.  getter로 값을 꺼낸다
3.  외부 서비스가 규칙을 판단한다
4.  setter나 update 메서드로 값을 다시 넣는다

이 구조는 처음에는 단순해 보이지만 장기적으로 취약하다.

------------------------------------------------------------------------

## 핵심 개념 1 캡슐화 붕괴

외부가 객체 내부 구조를 많이 알수록 캡슐화가 깨진다.

예

-   어떤 필드가 null인지 외부가 알아야 한다
-   어떤 순서로 상태를 바꿔야 하는지 외부가 알아야 한다
-   어떤 필드 조합이 유효한지 외부가 알아야 한다

이 상태가 되면 객체는 스스로 행동하지 못한다.

------------------------------------------------------------------------

## 핵심 개념 2 tell, don't ask

좋은 방식

객체에게 말한다.

예

-   포인트를 사용해라
-   예약을 확정해라
-   쿠폰을 사용 처리해라

나쁜 방식

객체에게 물어본다.

예

-   현재 포인트가 얼마니
-   그럼 내가 계산해서 다시 넣을게

즉 외부가 계산 주체가 되면 책임이 바깥으로 새어나간다.

------------------------------------------------------------------------

## 핵심 개념 3 데이터 중심 설계의 문제

문제는 단순히 setter가 있다는 것만이 아니다.

더 본질적인 문제

-   변경 이유가 외부 여러 곳으로 흩어진다
-   규칙이 한곳에 모이지 않는다
-   이름은 메서드인데 실제로는 필드 복사 함수가 된다
-   객체가 아닌 서비스가 도메인 규칙의 주인이 된다

------------------------------------------------------------------------

## 핵심 개념 4 의도를 드러내는 메시지

`update()`는 많은 것을 숨긴다.

반면 아래와 같은 이름은 의도를 드러낸다.

-   `confirm()`
-   `cancelByUser()`
-   `useCoupon()`
-   `expire()`
-   `changeSellableCapacity()`

좋은 메시지는 내부 구현을 몰라도 책임을 이해하게 만든다.

------------------------------------------------------------------------

## 핵심 개념 5 설계 품질은 변경에서 드러난다

새 정책이 추가될 때 무엇이 많이 수정되는지 보면 설계 품질이 드러난다.

좋지 않은 구조

-   서비스 if문 증가
-   타입 분기 증가
-   update 파라미터 증가
-   여러 객체의 내부 상태를 한 메서드가 함께 변경

좋은 구조

-   기존 협력은 유지된다
-   새 책임만 적절한 객체에 추가된다
-   서비스는 조율에 집중한다

------------------------------------------------------------------------

# 3장 4장 연결

3장

협력과 책임을 중심으로 객체를 본다.

4장

그 원칙을 지키지 않을 때, 데이터 중심 설계가 어떤 문제를 만드는지
보여준다.

즉 3장은 방향을 제시하고 4장은 반례를 통해 그 이유를 설명한다.

------------------------------------------------------------------------

# 스터디 토론 포인트

1.  서비스가 아니라 객체가 직접 책임져야 할 규칙은 무엇인가
2.  `update()`와 같은 메서드는 정말 행위인가, 아니면 setter 묶음인가
3.  어떤 기준으로 "이 책임은 객체 안으로 넣어야 한다"고 판단할 수 있을까
4.  역할은 잘 보이는데도 왜 여전히 타입 분기가 남는가
5.  tell-don't-ask를 적용하면 어떤 서비스가 가장 먼저 줄어들까

------------------------------------------------------------------------

# 실무 적용 포인트

많은 프로젝트에서 엔티티는 있고 객체지향은 없다.

문제

-   엔티티는 필드를 보관한다
-   서비스가 모든 규칙을 판단한다
-   스케줄러가 도메인 후처리까지 전부 수행한다

개선

도메인 객체가 의미 있는 메시지에 응답하도록 만든다.

예

재고 객체\
→ 예약 가능 여부와 수량 변경 책임

예약 객체\
→ 확정, 취소, 취소 사유 기록 책임

쿠폰 객체\
→ 사용, 만료, 복구 책임

포인트 객체\
→ 적립, 차감, 복구 책임

------------------------------------------------------------------------

새어나간다.
