## 데이터 중심적 사고
객체지향 설계에서 두 가지 방법을 이용해 시스템을 객체로 분할할 수 있다.

1. 상태를 분할의 중심축으로 삼는다
2. 책임을 분할의 중심축으로 삼는다

객체의 상태는 구현에 속함. => 구현은 불안정하기에 변하기 쉽다 즉 데이터에 초점을 맞추는 설계는 변경에 취약하다

데이터 중심의 설계는 객체가 포함해야 하는 데이터에 집중한다.<br>

만약 객체가 포함해야 하는 데이터가 뭐지? 라는 질문의 반복한다면 데이터 중심의 설계에 매몰돼 있을 확률이 높다<br>
(일단 나는 확실히 데이터 중심적으로 사고해왔다)

### 데이터 중심적 사고의 흐름
결국 데이터 중심적 사고는 객체 자신이 수행한다. <br>
협려 객체의 상태에 접근에서 자기가 수행한다는 것이다. 
- 영화 예메 Agency가 접근하는 상태들의 데이터에 접근해 본인이 영화 시간대, 할인정책, 가격 계산등을 모조리 수행한다.
- 마치 일병과 다를게 없다()

즉 단점으로
- 캡슐화 위반
- 높은 결합도, 낮은 응집도

### 책임 중심적 사고의 흐름
데이터 중심적 사고에서는 객체의 내부 상태를 밖으로 꺼내서 외부에서 계산하고 다시 집어넣는 흐름이였습니다.

책임 중심적 사고에서는 모두가 동등합니다. 묻지 말고 시킨다. <br>
객체에게 무언가를 해달라고 메시지를 보내서 해당 객체가 메시지를 받으면 자신의 역할을 수행해야 합니다.

즉 객체 스스로 자신의 상태를 처리할 수 있어야 한다.

이를 지키다보면 자연스럽게 캡슐화 또한 이루어지게 된다 = 변경에 용이한 설계
## 설계 트레이드 오프

### 캡슐화
객체의 내부 구현을 외부로부터 감춘다.

변경될 가능성이 높은 부분 = 구현 | 안정적인 부분 = 인터페이스 <br>
구현과 인터페이스를 분리하고 외부에서는 인터페이스에만 의존하도록 관계를 조절해라

캡슐화는 불안정한 부분과 안정적인 부분을 분리해서 변경의 영향을 통제할 수 있다. <br>
즉, 변경 가능성이 높은 부분(변경될 수 있는 어떠한 것이라도)을 객체 내부로 숨겨야 한다.

<br>

### 응집도와 결합도
- 응집도: 모듈에 포함된 내부 요소들이 연관돼 있는 정도
- 결합도: 의존성의 척도

좋은 설계란 = 높은 응집도와 낮은 결합도

변경의 관점에서 응집도 = 변경이 발생할 때 모듈 내부에서 발생하는 변경의 정도
- 하나의 변경을 수용하기 위해 모듈 전체가 함께 변경된다면 응집도가 높음
- 모듈의 일부만 변경된다면 응집도가 낮음

**결합도가 높아도 상관 없는 경우**
- 변경될 확률이 매우 적은 안정적인 모듈에 의존하는 것
- 표준 라이브러리에 포함된 모듈이나 stable한 프레임워크에 의존하는 경우

**캡슐화의 정도가 응집도와 결합도에 영향을 미친다**

## 데이터 중심 설계 왜 문제인가?
- 너무 이른 시기에 데이터에 관해 결정하도록 강요한다
- 협력이라는 문맥을 고려하지 않고 객체를 고립시킨 채 오퍼레이션을 결정한다

데이터 중심 설계는 해당 객체가 포함해야 하는 데이터가 무엇인가? 에 중점을 두기 때문에 처음부터 데이터에
관해 결정하도록 강요하며 너무 이른 시기에 **내부 구현에 초점을 맞추게 된다**.
