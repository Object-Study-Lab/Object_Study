# 오브젝트 1, 2장 정리 (스터디 Week1)

## 전체 핵심 요약

객체지향의 핵심은 클래스가 아니라 객체 간 협력이다.\
객체는 자신의 상태를 스스로 책임지고 행동해야 한다.\
외부 객체가 내부 데이터를 직접 조작하는 구조는 캡슐화를 깨고 결합도를
높인다.

설계는 다음 순서로 생각한다.

1.  어떤 협력이 필요한가
2.  어떤 책임이 필요한가
3.  그 책임을 누가 맡는가
4.  마지막에 객체와 구현을 결정한다

------------------------------------------------------------------------

# 1장 객체, 설계

## 1장의 핵심 주제

극장 예제를 통해 좋은 설계와 나쁜 설계를 비교한다.

초기 설계는 절차지향적인 구조다.\
Theater가 모든 객체의 내부 상태를 직접 확인하고 처리한다.

이 구조의 문제를 단계적으로 개선하면서 객체지향 설계가 무엇인지
보여준다.

핵심은 **책임 이동**이다.

------------------------------------------------------------------------

## 등장 객체

극장 예제의 주요 객체

-   Theater
-   TicketSeller
-   TicketOffice
-   Audience
-   Bag
-   Invitation
-   Ticket

이 예제의 핵심 질문

누가 관람객 입장을 책임져야 하는가

------------------------------------------------------------------------

## 초기 구조의 문제

초기 구조에서는 Theater가 대부분의 로직을 수행한다.

흐름

1.  Theater가 Audience를 받는다
2.  Audience 내부 Bag을 확인한다
3.  초대장이 있는지 확인한다
4.  있으면 티켓을 넣는다
5.  없으면 돈을 꺼내서 결제한다
6.  TicketOffice 상태를 변경한다

이 구조의 문제는 명확하다.

### 객체 자율성 없음

Audience는 자신의 가방을 가지고 있지만 실제 로직은 Theater가 수행한다.

즉 객체는 데이터를 가지고 있지만 행동 주체가 아니다.

### 캡슐화 붕괴

Theater가 알아야 하는 정보

-   Audience 내부 구조
-   Bag 존재 여부
-   초대장 존재 여부
-   결제 방식

객체 내부 구조가 외부에 노출되어 있다.

### 변경에 취약

예

-   할인 정책 추가
-   무료 입장 조건 추가
-   결제 방식 추가

Theater 수정 범위가 계속 커진다.

------------------------------------------------------------------------

## 핵심 개념 1 객체는 스스로 행동한다

외부가 객체 내부 데이터를 조작하면 안 된다.

잘못된 방식

외부 객체가 내부 상태를 직접 변경한다.

좋은 방식

객체에게 메시지를 보내고 객체가 스스로 처리한다.

객체는 단순 데이터 구조가 아니다.

객체 = 상태 + 행동

------------------------------------------------------------------------

## 핵심 개념 2 책임은 정보를 가진 객체에게 간다

객체는 자신이 가진 정보를 가장 잘 안다.

예

Audience\
→ 자신의 돈과 초대장 여부를 안다

Bag\
→ 티켓 보관 책임

TicketOffice\
→ 티켓 재고 관리

TicketSeller\
→ 판매 행위

외부 객체가 이를 모두 처리하면 설계가 깨진다.

------------------------------------------------------------------------

## 핵심 개념 3 캡슐화

객체 내부 상태는 숨긴다.

외부에서는 메시지만 사용한다.

장점

-   결합도 감소
-   변경 영향 최소화
-   유지보수 용이

------------------------------------------------------------------------

## 핵심 개념 4 협력 중심 설계

객체지향 설계의 시작점

객체가 아니다.

협력이다.

질문

-   어떤 기능이 필요한가
-   어떤 메시지가 필요한가
-   누가 그 메시지를 처리하는가

이 과정을 통해 객체가 결정된다.

------------------------------------------------------------------------

## 1장 핵심 정리

좋은 객체지향 설계 특징

-   객체가 스스로 행동한다
-   내부 상태는 숨긴다
-   책임이 적절히 분배된다
-   객체 간 메시지로 협력한다

------------------------------------------------------------------------

# 2장 객체지향 프로그래밍

## 핵심 주제

영화 예매 시스템을 통해 객체지향 설계를 구현한다.

주요 객체

-   Movie
-   Screening
-   Reservation
-   DiscountPolicy
-   DiscountCondition
-   Money

------------------------------------------------------------------------

## 예제 문제

영화 예매 시 가격은 할인 정책에 따라 달라진다.

예

-   특정 상영 순번 할인
-   특정 시간 할인
-   금액 할인
-   비율 할인

설계 목표

할인 정책이 변경되어도 시스템 구조는 안정적으로 유지한다.

------------------------------------------------------------------------

## 핵심 개념 1 협력 중심 객체

예매 시작점은 Movie가 아니다.

Screening이다.

왜냐하면 관객은 특정 상영 회차를 예매하기 때문이다.

Screening 책임

-   예매 처리
-   Movie에게 요금 계산 요청
-   Reservation 생성

------------------------------------------------------------------------

## 핵심 개념 2 Movie의 책임

Movie는 가격 정책을 알고 있다.

Screening은 Movie에게 요금 계산을 요청한다.

Movie 역할

-   기본 요금 보유
-   할인 정책 적용

------------------------------------------------------------------------

## 핵심 개념 3 할인 정책 추상화

할인 정책 종류

-   금액 할인
-   비율 할인

이 로직을 Movie 내부 if문으로 처리하면 문제 발생

문제

-   코드 복잡도 증가
-   변경 어려움
-   확장 어려움

해결

DiscountPolicy 추상화

------------------------------------------------------------------------

## 핵심 개념 4 다형성

DiscountPolicy를 기반으로 여러 정책 구현

예

AmountDiscountPolicy\
PercentDiscountPolicy

Movie는 구체 정책을 모른다.

Movie는 DiscountPolicy에게 메시지만 보낸다.

장점

-   정책 교체 가능
-   확장 용이
-   의존성 감소

------------------------------------------------------------------------

## 핵심 개념 5 할인 조건 분리

할인 조건도 객체로 분리한다.

예

-   순번 조건
-   시간 조건

이유

정책과 조건은 변경 이유가 다르다.

단일 책임 원칙 적용

------------------------------------------------------------------------

## 핵심 개념 6 합성

Movie는 DiscountPolicy를 상속받지 않는다.

객체로 가진다.

구조

Movie → DiscountPolicy

장점

-   유연한 설계
-   정책 교체 쉬움
-   결합도 감소

------------------------------------------------------------------------

## 핵심 개념 7 추상화 의존

좋은 설계

구현이 아니라 추상화에 의존한다.

Movie 의존 대상

DiscountPolicy

구현

AmountDiscountPolicy\
PercentDiscountPolicy

이 구조는 변경에 강하다.
