# 학습 정리

## 객체지향 전환
1. 어떤 클래스가 필요한지를 고민하기 전에 어떤 객체들이 필요한지 고민하라. 클래스는 공통적인 상태와 행동을 공유하는 객체들을 추상화한 것이다


2. 객체를 독립적인 존재가 아니라 기능을 구현하기 위해 협력하는 공동체의 일원으로 봐야 한다.

## 자율적인 객체
객체는 상태와 행동을 함께 가진다. <br>
즉, 객체는 스스로 판단하고 행동하는 자율적인 존재이다                  

어떻게 하면 객체를 자율적이게 만들 수 있을가?
- 접근 제어를 통해 캡슐화
- 외부의 간섭을 최소화 한다
- 객체의 상태는 숨기고 행동만을 외부에 공개한다

객체를 더 자율적인 존재로 만들기 위해서는 외부의 간섭을 최소화해야 한다.(외부로부터 영향을 덜받음) = 변경의 영향으로부터 벗어남

필요한 부분만 공개하고 나머지는 숨긴다. 이를 사용하는 측에서는 내부의 구현은 무시한 채 인터페이스만 알고 있어도 클래스를 사용할 수 있다.

## 컴파일 타임 의존성과 런타임 의존성
상속 및 인터페이스를 사용 시 추상화된 타입을 사용할 수 있다. <br>

추상화된 타입을 받게되면 컴파일 타임에는 실제로 어떤 객체를 의존하는지 알 수 없다. 즉 코드의 의존성과 실행 시점의 의존성이 서로 다를 수 있다.

코드의 의존성과 실행 시점의 의존성이 다르면 코드를 이해하기 어려워진다. 하지만 코드는 더 유연해지고 확장 가능해진다.
또 구현 객체를 쉽게 변경할 수 있으므로 테스트 또한 용이해지게 된다. 

정리하자면 이 또한 트레이드 오프이다. <br>
유연성을 억제하면 코드를 이해하고 디버깅이 쉬워지지만 재사용과 확장성이 낮아진다.

## 다형성
객체지향 프로그램의 컴파일 시간 의존성과 런타임 의존성이 다를 수 있다는 사실을 기반으로 한다. 

다형성이란 동일한 메시지를 수신했을 때 객체의 타입에 따라 다르게 응답할 수 있는 능력을 의미한다.
실행 시점에 바인딩하는 것을 지연 바인딩 또는 동적 바인딩이라고 한다.

## 추상클래스 vs 인터페이스
이 부분은 꽤 재밌다. <br>
추상 클래스를 통해서도 추상화된 타입을 사용할 수 있다.

하지만 추상클래스를 사용하다보면 부모 클래스의 코드가 변경되면 그 영향은 자식까지 전파되는 문제가 있다. <br>
즉 OCP를 위반하게 된다. 부모의 코드가 변경되면 이를 상속하는 자식 클래스까지 영향이 갈 확률이 높다. <br>
또 캡슐화를 위반한다. 부모 클래스의 구현이 자식 클래스에게 노출된다.

그럼 추상 클래스보단 무조건 인터페이스인가? <br>

이상적으로는 인터페이스를 사용한 설계가 더 좋지만 그렇다고 해서 추상 클래스가 무조건 나쁘다는 것은 아니다.
코드 재사용이 목적이라면 추상 클래스도 합리적인 선택이다. 

AmountDiscountPolicy와 PercentDiscountPolic가 조건 로직을 공통으로 재사용해야 하기 때문에 
Template Method 패턴과 함께 추상 클래스를 사용했다. 
이처럼 상속과 합성을 함께 조합해서 많이 사용한다.

## 합성
합성
인터페이스에 정의된 메시지를 통해서만 코드를 재사용하는 방법을 합성이라고 한다.

```kotlin
// 상속
class Movie : DiscountPolicy()

// 합성
class Movie(
    private val discountPolicy: DiscountPolicy
)
```
상속은 부모 클래스의 내부 구현을 자식 클래스와 컴파일 시점에 강하게 결합시킨다. 

반면 합성은 인터페이스를 통해 약하게 결합되기 때문에 내부 구현이 바뀌어도 외부에 영향을 주지 않는다.


코드 재사용을 위해서는 상속보다 합성을 선호하는 것이 좋다. 다만 다형성을 위한 타입 공유가 목적이라면 상속과 합성을 함께 조합해서 사용할 수밖에 없다.
